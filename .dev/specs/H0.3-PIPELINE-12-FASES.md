**Épico:** E0 — Realinhamento Excalidraw
**Prioridade:** P0 (crítico)
**Estimativa:** M
**Risco:** Médio (migração de dados existentes)

**PRD:** `.dev/production/PRD-v3.0-RECONCILIADO.md` — Seção 8

---

## 1. Objetivo

Expandir o pipeline de M&A dos 5 marcos jurídicos atuais (`teaser → nda → nbo → spa → closed_lost`) para as 12 fases operacionais definidas no Excalidraw, transformando o pipeline de um tracker jurídico simplificado em um kanban operacional completo de deal management.

---

## 2. Diagnóstico — Estado Atual vs Excalidraw

### Pipeline atual (5 marcos)

```
teaser → nda → nbo → spa → closed_lost
```

Enum em `src/types/database.ts`:

```typescript
project_status: "teaser" | "nda" | "nbo" | "spa" | "closed_lost"
```

Labels:

```typescript
PROJECT_STATUS_LABELS = {
  teaser: 'Teaser',
  nda: 'NDA',
  nbo: 'NBO',
  spa: 'SPA',
  closed_lost: 'Fechado-Perdido',
}
```

### Pipeline Excalidraw (12 fases)

```
Screening → Teaser → NDA → CIM/DFs → IoI → Management Meetings → NBO → DD/SPA → Signing → CPs → Closing → Disclosure
```

Plus saídas laterais: **Fechado** (sucesso) e **Perdido** (saída de qualquer fase).

### Mapeamento 5 → 12


| Fase atual    | Fases Excalidraw correspondentes       |
| ------------- | -------------------------------------- |
| —             | **Screening** (NOVA — antes do Teaser) |
| `teaser`      | **Teaser** (mantido)                   |
| `nda`         | **NDA** (mantido)                      |
| —             | **CIM/DFs** (NOVA)                     |
| —             | **IoI** (NOVA)                         |
| —             | **Management Meetings** (NOVA)         |
| `nbo`         | **NBO** (mantido)                      |
| `spa`         | **DD/SPA** (expandido)                 |
| —             | **Signing** (NOVA)                     |
| —             | **CPs** (NOVA)                         |
| —             | **Closing** (NOVA)                     |
| —             | **Disclosure** (NOVA)                  |
| `closed_lost` | **Fechado** + **Perdido** (separados)  |


**Resultado:** 3 fases existentes mantidas, 1 expandida (spa → DD/SPA), 1 separada (closed_lost → Fechado + Perdido), 7 fases novas.

## 4. Matriz de Transição

### Regras de avanço

Transições permitidas (pode avançar para a próxima fase ou pular para saída):

```typescript
const ALLOWED_TRANSITIONS: Record<ProjectStatus, ProjectStatus[]> = {
  screening:           ['teaser', 'closed_lost'],
  teaser:              ['nda', 'closed_lost'],
  nda:                 ['cim_dfs', 'closed_lost'],
  cim_dfs:             ['ioi', 'closed_lost'],
  ioi:                 ['management_meetings', 'closed_lost'],
  management_meetings: ['nbo', 'closed_lost'],
  nbo:                 ['dd_spa', 'closed_lost'],
  dd_spa:              ['signing', 'closed_lost'],
  signing:             ['cps', 'closed_lost'],
  cps:                 ['closing', 'closed_lost'],
  closing:             ['disclosure', 'closed_won', 'closed_lost'],
  disclosure:          ['closed_won', 'closed_lost'],
  closed_won:          [],  // Estado terminal
  closed_lost:         [],  // Estado terminal
}
```

### Gates condicionais (mantidos do E4)


| Transição      | Condição                             |
| -------------- | ------------------------------------ |
| → `nda`        | NDA request aceita pelo ativo        |
| → `nbo`        | NBO formal submetida                 |
| → `dd_spa`     | Acesso ao VDR Due Diligence liberado |
| → `signing`    | Documentos legais finalizados        |
| → `closed_won` | SPA assinado e condições cumpridas   |


### Liberação progressiva do MRS (mantida)


| Evento        | MRS liberado    |
| ------------- | --------------- |
| NDA assinado  | Passos 1 e 2    |
| NBO submetida | Passos 3 e 4    |
| SPA/Closing   | Status: Fechado |

### 5.2 Kanban UI

**Layout:** 12 colunas horizontais com scroll horizontal. Cada coluna:

- Header: nome da fase + contador `(n)`
- Cards: nome do investidor (clicável → perfil)
- Drag & drop entre fases adjacentes (respeitando matriz de transição)

**Controles (Excalidraw):**

- Toggle: `Pipeline` / `Lista` (kanban vs tabela)
- Busca por investidor
- Filtros por fase
- Ações em massa
- Permissões
- Members
- Share

**Saídas laterais:**

- `Fechado` e `Perdido` como áreas separadas abaixo ou ao lado do kanban
- Card pode ser arrastado para `Perdido` de qualquer fase

### 5.3 Responsividade

12 colunas não cabem em tela — necessário:

- Scroll horizontal com indicador de posição
- Colunas com largura mínima fixa (ex: 200px)
- Alternativa: visão compacta colapsando fases sem cards

## 6. Migration Supabase

```sql
-- 1. Adicionar novos valores ao enum
ALTER TYPE project_status ADD VALUE IF NOT EXISTS 'screening';
ALTER TYPE project_status ADD VALUE IF NOT EXISTS 'cim_dfs';
ALTER TYPE project_status ADD VALUE IF NOT EXISTS 'ioi';
ALTER TYPE project_status ADD VALUE IF NOT EXISTS 'management_meetings';
ALTER TYPE project_status ADD VALUE IF NOT EXISTS 'dd_spa';
ALTER TYPE project_status ADD VALUE IF NOT EXISTS 'signing';
ALTER TYPE project_status ADD VALUE IF NOT EXISTS 'cps';
ALTER TYPE project_status ADD VALUE IF NOT EXISTS 'closing';
ALTER TYPE project_status ADD VALUE IF NOT EXISTS 'disclosure';
ALTER TYPE project_status ADD VALUE IF NOT EXISTS 'closed_won';

-- 2. Migrar dados existentes
-- spa → dd_spa (rename semântico)
UPDATE projects SET status = 'dd_spa' WHERE status = 'spa';

-- closed_lost permanece (pode ser reclassificado manualmente depois)

-- 3. Registrar migração no audit_log
INSERT INTO audit_logs (event_type, payload)
VALUES ('pipeline.migration_5_to_12', jsonb_build_object(
  'description', 'Expandir pipeline de 5 para 12 fases conforme Excalidraw',
  'migrated_at', now(),
  'spa_to_dd_spa', (SELECT count(*) FROM projects WHERE status = 'dd_spa')
));
```

**Nota sobre enum PostgreSQL:** `ALTER TYPE ADD VALUE` não é transacional. Cada ADD VALUE deve ser em statement separado ou fora de transaction block.

## 10. Riscos e Mitigações


| Risco                                         | Mitigação                                                   |
| --------------------------------------------- | ----------------------------------------------------------- |
| ALTER TYPE ADD VALUE não é transacional       | Executar fora de transaction block, uma statement por vez   |
| 12 colunas não cabem na tela                  | Scroll horizontal + visão compacta                          |
| Migração `spa` → `dd_spa` pode afetar queries | Search & replace em todo o codebase                         |
| Gates existentes podem conflitar              | Manter gates atuais, adicionar novos apenas onde necessário |

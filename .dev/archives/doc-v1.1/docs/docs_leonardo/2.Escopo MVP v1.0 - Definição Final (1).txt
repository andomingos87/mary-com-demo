## **User Journey 1: Investidor → Deal Flow Qualificado**

### **Objetivo:** Acesso a deal flow estruturado com matching automático baseado em teses

```
Onboarding → Thesis Creation → Automated Radar → Pipeline Management → Deal Closing
```

### **Etapas Detalhadas:**

**1. Onboarding & Authentication**
- **Rota:** `/:investorname/` (slug criado no callback)
- **MFA obrigatório:** E-mail + WhatsApp
- **Campos mínimos:** Site e CNPJ > AI extrai tudo que for possível, antes de abrir campos: tipo de Investidor (Ver Perfis no anexo "Arquitetura Mestre - Mary v2"), AUM, dados institucionais, responsáveis.
- **Alerta persistente:** "Configure sua primeira tese para começar a receber oportunidades"
- **Anexo "Fluxos de Onboarding e Verificação":** Essencial para filtrar Investidores e Advisors sério, profissionais e relevantes, para não poluir nem desacreditar a Mary.



**2. Thesis Creation (`/:investorname/thesis`)**
- **Mary Taxonomy Integration:** Ver anexo "Mary Taxonomy v1"
- **Campos críticos:**
- **IA Assistant:** Pode fazer sugestão inicial de toda tese, com base no perfil do investidor
  - Nome da tese
  - Setores alvo (multi-seleção) > Mary Taxonomy
  - Modelos de negócio (SaaS, Marketplace, B2B Services, etc.)
  - Geografia (países/regiões)
  - Faixa de investimento (Ticket size) (USD min/max)
  - Tipo participação (integral/controle/minoritária relevante/minoritária)
  - Faixa receita (USD min/max) alvo
  - Faixa EBITDA (USD min/max) alvo
  - Critérios específicos (founder-led, carve-out, etc.)
  - Restrições/exclusões
- **IA Assistant:** Sugere refinamentos baseados em dados de mercado
- **Resultado:** Tese ativa alimenta matching automático

**3. Automated Opportunities Radar (`/:investorname/opportunities`)**
- **Interface:** Cards de ativos com matching score + explicação
- **Dados por card:** Codenome, segmento, faturamento, EBITDA, overview confidencial, objetivo, Readiness Score
- **Matching Algorithm:** Score 0-100 com "why it matches"
  - Fit setorial/modelo (40%)
  - Tamanho/ticket (20%) 
  - Geografia (10%)
  - Objetivo vs. tese (10%)
  - Readiness/cobertura L2+ (20%)
- **Filtros persistentes:** Readiness Score, setor, ticket, localização, status pipeline
- **CTA principal:** "Acessar Teaser" → move para pipeline

**4. Pipeline Management (`/:investorname/pipeline`)**
- **Controle:** Investidor controla avanço via CTAs
- **Etapas e gatilhos:**
  - **Teaser** → CTA: "Assinar NDA"
  - **NDA** → Gatilho: Upload NDA assinado
  - **Pré-DD** → CTA: "Indicar Interesse"  
  - **IoI** → Gatilho: Upload IoI assinada
  - **Management Meetings** → CTA: "Avançar para NBO"
  - **NBO** → Gatilho: Upload NBO assinada
  - **DD/SPA** → CTA: "Avançar para Signing"
  - **Signing** → Gatilho: Upload SPA assinado
  - **CP's** → CTA: "CP's Concluídas"
  - **Closing** → Gatilho: "Closing Concluído"
  - **Disclosure** → CTA: "Deal Divulgado"
  - **Declinado** → CTA: "Recusar"
- **Sincronização:** Pipeline espelhado com área do ativo
- **Notificações:** WhatsApp + e-mail + in-app para cada mudança

**5. VDR Access & Q&A**
- **Teaser:** Acesso direto via link rastreável
- **NDA Required:** Para criação do VDR específico do investidor
- **Copy-on-Share:** VDR copia snapshot do Asset VDR após NDA
- **Q&A Management:** Threads por documento com estados (Open, Waiting, Resolved)
- **Isolation:** VDR completamente separado entre investidores

## **User Journey 2: Ativo → Exposição Qualificada**

### **Objetivo:** Estruturar dados, gerar materiais automáticos e receber interesse qualificado

```
Onboarding → Data Structuring → Project Creation → AI Content → Investor Matching → Deal Management
```

### **Etapas Detalhadas:**

**1. Company Onboarding (`/:assetname/`)**
- **CNPJ Integration:** Auto-preenchimento dados públicos
- **Campos mínimos:** Razão social, nome fantasia, site, setor (Mary Taxonomy), receita 12m, EBITDA 12m, objetivo
- **Readiness inicial:** Cálculo automático com dados L1

**2. Data Structuring Phase**
- **Asset VDR (`/:assetname/assetvdr`):** Árvore padrão + uploads customizados
- **L1 Tagging:** Todos os uploads iniciais marcados como "Raw"
- **Financials (`/:assetname/financials`):** IA padroniza DRE/BS/FC
- **Cap Table (`/:assetname/cap-table`):** Estrutura societária + versionamento
- **KPIs (`/:assetname/kpis`):** Métricas por segmento (receita, EBITDA, margem, churn/LTV/GMV)

**3. Mary Readiness Score (`/:assetname/readiness`)**
- **Cálculo ponderado:** L1=0.3, L2=0.7, L3=1.0 por tipo de dado
- **Categorias:** Financeiro, operacional, jurídico, estratégico
- **Roadmap automático:** IA sugere gaps a atacar para melhorar score
- **Integração:** Score impacta ranking no matching de investidores

**4. Project Creation (`/:assetname/projects/`)**
- **Codename:** Sugestão automática "MARY-ABCD" + editável
- **Objetivo:** Venda, captação, fusão
- **IA Content Generation:**
  - **Teaser:** Formato chat scrollável com dados estruturados
  - **Valuation:** Múltiplos automáticos + metodologia explicada
  - **CIM:** Documento completo baseado no Asset VDR
- **Links rastreáveis:** Sem export PDF, apenas compartilhamento com tracking

**5. Investor Matching (`/:assetname/projects/:codename/investors`)**
- **Lista automática:** Matching reverso baseado em teses cadastradas
- **Manual selection:** Busca e filtros para seleção adicional
- **Outreach:** Mensagens in-app para investidores selecionados
- **Pipeline sync:** Criação automática de cards no pipeline do investidor

**6. VDR Management per Investor**
- **Trigger:** Criação automática após validação de NDA
- **Rota isolada:** `/:assetname/projects/:codename/investors/:investorname/vdr`
- **Q&A bidirectional:** Upload de RFI pelo investidor + respostas do ativo
- **L2 Elevation:** Advisor pode elevar documentos durante o processo

## **User Journey 3: Advisor → Curadoria L2 e Eficiência**

### **Objetivo:** Gestão eficiente de mandatos com curadoria de dados L1→L2

```
Authorization → Conflict Check → Project Access → L2 Validation → Collaborative Management
```

### **Etapas Detalhadas:**

**1. Authorization & Mandate Registration**
- **Registro inicial:** Contrato/mandato registrado no início do processo
- **Conflict prevention:** Sistema impede acesso a ambos os lados do mesmo projeto
- **RBAC Assignment:** Owner/Admin define permissões granulares (Member com permissão L2)

**2. Project Access (Contextual)**
- **Sell-side:** Acesso apenas à área do ativo (`/:assetname/*`)
- **Buy-side:** Acesso apenas à área do investidor (`/:investorname/*`)
- **Lista consolidada:** Visão agregada em `/advisor/projects` (v1.5)

**3. L2 Data Validation**
- **Curadoria:** Revisão de documentos e campos L1 → L2
- **Tracking:** Metadados completos (validated_by, timestamp, hash, comments)
- **Padronização:** Formatação e estruturação de materiais
- **Diff viewer:** Comparação entre versões L1 e L2

**4. Collaborative Tools**
- **Task creation:** Dentro dos projetos com atribuição @users
- **Q&A participation:** Threads no VDR com expertise técnica
- **Document review:** Teaser, CIM, valuation com sugestões de melhoria

## **Arquitetura de Integração Cross-Profile**

### **Data Flow com L1/L2/L3**
```
Ativo (L1 raw data) → Advisor (L2 curation) → Investidor (qualified view)
                           ↓
Pipeline sync ← → Task management ← → Notifications ← → Mary AI Context
```

### **Security & Confidentiality**
- **VDR Isolation:** Cada investidor = namespace completamente separado
- **RBAC Enforcement:** Advisor nunca acessa ambos os lados simultaneamente
- **Mary AI Context:** Limitada ao workspace_id + project_id da sessão ativa
- **Audit Trail:** Logs granulares de acesso, visualização, modificação

### **Automation Triggers**
- **Matching Engine:** Tese ↔ Ativo com score explicável em tempo real
- **Pipeline Advancement:** CTAs → notificações → sync bidirecional
- **Readiness Scoring:** L1/L2/L3 validation → MRS update → ranking impact

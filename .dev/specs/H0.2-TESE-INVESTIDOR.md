## 1. Objetivo

Adicionar campos faltantes na tese de investimento conforme Excalidraw. A divergência principal é que **ROB (Receita Operacional Bruta) e Cheque (ticket) são conceitos distintos** no Excalidraw, mas na implementação atual o `ticketMin/Max` está sendo usado para ambos, causando confusão semântica.

## 3. Novo Schema `ThesisCriteria`

```typescript
interface ThesisCriteria {
  // === Passo 1 — Criar Tese ===
  sectors?: string[]                   // ✅ Mantido — Setores-alvo (MAICS)
  targetAudience?: string[]            // NOVO — Público-alvo: ['B2B', 'B2C', 'B2B2C', 'B2G']
  geo?: string[]                       // ✅ Mantido — Regiões (GeographySelector)
  robMin?: number | null               // NOVO — ROB mínimo (moeda)
  robMax?: number | null               // NOVO — ROB máximo (moeda)
  ebitdaPercentMin?: number | null     // NOVO — EBITDA % mínimo
  ticketMin?: number | null            // ✅ Mantido — Cheque mínimo (renomear label)
  ticketMax?: number | null            // ✅ Mantido — Cheque máximo (renomear label)

  // === Passo 2 — Refinar Tese ===
  stage?: string[]                     // ✅ Mantido — Estágio (Seed, Série A, etc.)
  operationType?: string[]             // NOVO — Tipo de operação
  exclusionCriteria?: string           // NOVO — Critérios de exclusão (texto livre)
  additionalInfo?: string              // NOVO — Informações adicionais (texto livre)

  [key: string]: unknown               // Mantido para flexibilidade
}
```

### Diferenciação ROB vs Cheque


| Conceito   | Campo                     | Significado                                                                   |
| ---------- | ------------------------- | ----------------------------------------------------------------------------- |
| **ROB**    | `robMin` / `robMax`       | Receita Operacional Bruta da empresa-alvo (critério de filtragem por tamanho) |
| **Cheque** | `ticketMin` / `ticketMax` | Valor que o investidor está disposto a investir/aportar                       |


**Exemplo:** Um investidor pode buscar empresas com ROB de R$10M-50M mas estar disposto a investir (cheque) de R$5M-20M. São métricas diferentes.

## 4. Opções dos Novos Campos

### Público-alvo (`targetAudience`)

Multi-select com opções:

- B2B (Business to Business)
- B2C (Business to Consumer)
- B2B2C (Business to Business to Consumer)
- B2G (Business to Government)

### Tipo de operação (`operationType`)

Multi-select com opções:

- Participação minoritária
- Participação majoritária
- Venda integral (100%)
- Buyout
- Growth equity
- Venture capital
- Outro

### Estágio (`stage`) — expandir opções

Opções atuais + novas conforme Excalidraw:

- Pre-seed
- Seed
- Série A
- Série B
- Série C+
- Growth
- Late Stage
- Lucrativa / Mature
- Consolidada

## 6. Migração de Dados

### Dados existentes em `criteria` JSON

Os dados existentes nas teses já criadas devem ser preservados:

```typescript
// Migração automática (pode ser feita em runtime ou via migration)
function migrateCriteria(old: OldThesisCriteria): ThesisCriteria {
  return {
    ...old,
    // ticketMin/Max mantidos como cheque (semântica corrigida apenas no label)
    // Novos campos ficam undefined/null (preenchimento posterior)
    robMin: null,
    robMax: null,
    ebitdaPercentMin: null,
    targetAudience: [],
    operationType: [],
    exclusionCriteria: '',
    additionalInfo: '',
  }
}
```

**Nota:** Como `criteria` é JSON flexível, não há necessidade de migration SQL. Os novos campos simplesmente não existem nas teses antigas e o código trata `undefined` graciosamente.

## 7. Impacto no Matching (Radar)

Os novos campos devem ser considerados no matching do Radar (`src/lib/actions/radar.ts`):


| Campo               | Matching                                                           |
| ------------------- | ------------------------------------------------------------------ |
| `robMin/Max`        | Comparar com `organizations.onboarding_data.rob12Months` do ativo  |
| `ebitdaPercentMin`  | Comparar com `onboarding_data.ebitdaPercent` do ativo              |
| `targetAudience`    | Comparar com `onboarding_data.businessModel` do ativo              |
| `operationType`     | Comparar com `onboarding_data.projectObjective.type` do ativo      |
| `exclusionCriteria` | Texto livre — não entra no matching automático (apenas referência) |


**Dependência:** H0.1 deve ser implementado primeiro para que os campos do ativo existam para matching.
